# CAW Pact API Key Fallback Report

## Problem

The app returned:

```text
CAW SDK validation error (422): Active Pact did not return a pact-scoped API key for transaction execution.
```

This error was generated by the local server before calling `TransactionsApi.transferTokens`.

## Root Cause

The code required `PactsApi.getPact(AGENT_WALLET_PACT_ID)` to return `result.api_key`.

For the current active Pact, CAW did not return `api_key` to this server request. That may happen depending on the principal/API key visibility model.

Because the local code treated missing `api_key` as fatal, real transfer execution stopped before CAW could evaluate the transfer payload.

## Fix

Updated `lib/wallets/cawServer.ts`:

- Continue to call `PactsApi.getPact(AGENT_WALLET_PACT_ID)`.
- Prefer returned Pact-scoped `api_key` when available.
- If the active Pact does not return `api_key`, safely fall back to the existing server-only `AGENT_WALLET_API_KEY`.
- Do not expose either key to the frontend.
- Do not include API keys in logs, previews, errors, or reports.

This removes the local `missing_pact_api_key` blocker while preserving the server-side CAW execution path.

## Tests Updated

Updated `tests/cawIntegration.test.ts` to cover:
